# План тестирования YDB в VectorDBBench и анализ достаточности кода

Документ описывает конкретный план запуска бенчмарков векторного поиска для
сравнения **YDB** с популярными векторными СУБД, с акцентом на **горизонтальное
масштабирование YDB**. Вторая часть документа — проверка того, что текущая
реализация поддержки YDB (`vectordb_bench/backend/clients/ydb/`) достаточна для
выполнения этого плана, и перечень обнаруженных недоработок.

---

## 1. Цели и методология

YDB — распределённая СУБД, и её главное отличие от большинства движков из
лидерборда (pgvector, Qdrant, Milvus standalone, Elasticsearch single-node и
т. п.) — это **горизонтальное масштабирование**: данные разбиваются на таблетки
(DataShard), которые распределяются по узлам кластера, а SDK-драйвер сам
балансирует запросы. Поэтому план строится вокруг двух осей:

1. **Сопоставимость** — YDB и конкуренты прогоняются на одинаковых датасетах,
   кейсах и при одинаковой нагрузке клиента (одни и те же `--case-type`, `--k`,
   `--num-concurrency`, `--concurrency-duration`), чтобы результаты можно было
   класть рядом на лидерборде и сравнивать по QPS, p99-latency и recall.
2. **Масштабируемость** — YDB прогоняется на кластерах **разного размера**
   (например 1 → 3 → 9 узлов) на одном и том же кейсе, чтобы измерить, как
   растут QPS/throughput и как ведёт себя latency и recall при добавлении узлов.

Ключевой принцип VectorDBBench: «система под тестом разворачивается как угодно,
клиентские параметры можно тюнить». Размер кластера YDB — это **операционный**
параметр развёртывания, его сам бенчмарк не задаёт; различия между прогонами
фиксируются через `--db-label` (рекомендуется кодировать туда размер/тип
инстанса, как это делается для эталонных результатов проекта).

### Метрики, которые собирает фреймворк (на кейс)

- `insert_duration`, `optimize_duration`, `load_duration` (загрузка + построение индекса);
- `recall`, `ndcg`, `serial_latency_p99/p95` (серийный поиск, 1 поток);
- `qps` (максимум по списку конкуррентностей), а также по каждой конкуррентности —
  `conc_qps_list`, `conc_latency_p99/p95/avg_list`.

### Базовые предпосылки

- Клиент бенчмарка во всех прогонах одинаковый (тот же хост/CPU/сеть), желательно
  в той же зоне, что и сервер — иначе сетевой RTT исказит latency.
- Метрика расстояния выбирается автоматически из датасета (см. §5, `assembler.py`),
  отдельно её задавать не нужно.
- Каждый кейс изолируется собственной таблицей (`--table-name` или
  авто-имя по case-type, см. `task_runner._case_unique_collection_name`).

## 2. Подготовка окружения

### 2.1. Конкуренты (минимальный сопоставимый набор)

Берём системы, которые уже есть в проекте и покрывают разные классы движков:

| Класс | СУБД | Команда CLI |
|------|------|-------------|
| PostgreSQL + HNSW | pgvector | `vectordbbench pgvectorhnsw` |
| PostgreSQL + IVF/RaBitQ | VectorChord | `vectordbbench vectorchordrq` |
| Специализированный ANN | Milvus | `vectordbbench milvushnsw` |
| Специализированный ANN | Qdrant | `vectordbbench qdrant` |
| Поисковый движок | Elasticsearch/OpenSearch | `vectordbbench elasticcloudhnsw` / `awsopensearch` |
| Распределённый OLTP/HTAP + вектор | **YDB** | `vectordbbench ydb` |

Для честного сравнения **«масштабируемость против масштабируемости»** в качестве
оппонентов берём системы, которые тоже умеют шардироваться: Milvus (distributed),
OpenSearch (shards/replicas), Qdrant (sharding). Single-node движки (pgvector)
оставляем как «однопроцессный baseline».

### 2.2. YDB кластеры

Готовим **минимум 3 конфигурации** одного и того же кластера, отличающиеся только
числом узлов хранения/вычисления, например:

- `ydb-1n` — 1 узел;
- `ydb-3n` — 3 узла;
- `ydb-9n` — 9 узлов.

Подключение во всех случаях — через endpoint discovery (драйвер сам балансирует):

```shell
export YDB_ENDPOINT=grpc://<lb-or-any-node>:2136
export YDB_DATABASE=/Root/db1
# при необходимости: YDB_USER / YDB_PASSWORD или SDK-креды
```

> Важно (см. §6, недоработка G): чтобы данные реально распределялись по узлам,
> таблица должна быть разбита на несколько таблеток. На момент написания клиент
> создаёт таблицу **без** настроек партиционирования — это нужно учесть при
> подготовке окружения (предсоздать/донастроить таблицу) либо доработать код.

## 3. Матрица датасетов и кейсов

Используем штатные кейсы (все они уже заявлены как поддерживаемые для YDB):

| Ось | Кейсы | Назначение |
|-----|-------|-----------|
| Размер (768d, Cohere) | `Performance768D1M` → `Performance768D10M` | масштаб по объёму |
| Размер (1536d, OpenAI) | `Performance1536D500K` → `Performance1536D5M` | большая размерность |
| Малый (для smoke) | `Performance1536D50K` | быстрый прогон/отладка |
| Int-фильтр | `Performance768D1M1P` / `...99P`, `Performance1536D500K1P` / `...99P` | фильтрация по диапазону id |
| Label-фильтр | `LabelFilterPerformanceCase` (+ `--dataset-with-size-type`) | фильтрация по равенству метки |

Свип параметров YDB-индекса для кривой recall/QPS:
`--kmeans-tree-search-top-size` ∈ {1, 4, 10, 16, 32}, опционально `--levels`/`--clusters`.

## 4. Конкретный порядок запуска

Везде ниже подразумевается активированное окружение
(`pip install -e '.[ydb]'`) и выставленные `YDB_ENDPOINT`/`YDB_DATABASE`.
Конкуррентность и длительность держим **одинаковыми** для YDB и конкурентов.

### Фаза 0 — Smoke / валидация конвейера (минуты)

Цель: убедиться, что загрузка → построение индекса → поиск работают end-to-end,
прежде чем тратить часы на большие датасеты.

```shell
vectordbbench ydb \
  --case-type Performance1536D50K \
  --db-label ydb-1n-smoke \
  --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,10 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10
```

Проверяем в результатах: `recall` разумный (> 0.9 при top-size 10 на 50K),
`load_duration` конечный, `qps` ненулевой. Также прогнать unit/инт-тесты:
`pytest tests/test_ydb.py` (интеграционные требуют живого YDB).

### Фаза 1 — Однокластерный baseline (medium датасеты)

Цель: получить опорную точку YDB на 1 узле для последующего сравнения с
конкурентами и с многоузловыми прогонами.

```shell
# 768d / 1M (Cohere)
vectordbbench ydb --case-type Performance768D1M \
  --db-label ydb-1n --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10

# 1536d / 500K (OpenAI)
vectordbbench ydb --case-type Performance1536D500K \
  --db-label ydb-1n --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10
```

### Фаза 2 — Горизонтальное масштабирование (главная фаза для YDB)

Цель: на **одном и том же кейсе** замерить прирост QPS/latency при росте числа
узлов. Меняется только кластер и `--db-label`; всё остальное идентично.

```shell
# Кластер 1 узел
vectordbbench ydb --case-type Performance768D10M --db-label ydb-1n \
  --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,10,20,40,60,80,100,120 --concurrency-duration 60 \
  --kmeans-tree-search-top-size 10

# Кластер 3 узла (переключить YDB_ENDPOINT/кластер) — те же параметры
vectordbbench ydb --case-type Performance768D10M --db-label ydb-3n \
  --skip-drop-old --skip-load --search-serial --search-concurrent \
  --num-concurrency 1,10,20,40,60,80,100,120 --concurrency-duration 60 \
  --kmeans-tree-search-top-size 10

# Кластер 9 узлов — те же параметры, --db-label ydb-9n
```

Замечания:
- На фазе масштабирования данные грузим **один раз на кластер** (`--drop-old --load`
  на первом запуске для данного кластера), а повторные конфигурации
  поиска запускаем с `--skip-drop-old --skip-load`, чтобы не перегружать данные.
- Доводим список конкуррентностей до значений, где QPS выходит на плато — именно
  «потолок» при разном числе узлов показывает масштабируемость.
- Для теста масштабирования **записи/throughput** отдельно снимаем
  `insert_duration` (а лучше — кейс `CloudInsertCase`, см. ниже) на 1/3/9 узлах.

### Фаза 3 — Масштабирование по объёму данных

Цель: показать, как YDB держит рост датасета (фиксируем кластер, например 3 узла).

```shell
for CASE in Performance768D1M Performance768D10M; do
  vectordbbench ydb --case-type $CASE --db-label ydb-3n \
    --drop-old --load --search-serial --search-concurrent \
    --num-concurrency 1,10,20,40,60,80 --concurrency-duration 60 \
    --kmeans-tree-search-top-size 10
done
```

### Фаза 4 — Фильтрация

```shell
# Int-фильтр (диапазон id): 1% и 99%
vectordbbench ydb --case-type Performance768D1M1P  --db-label ydb-3n --drop-old --load --search-serial --search-concurrent
vectordbbench ydb --case-type Performance768D1M99P --db-label ydb-3n --drop-old --load --search-serial --search-concurrent

# Label-фильтр (равенство метки) — нужен --dataset-with-size-type
vectordbbench ydb --case-type LabelFilterPerformanceCase \
  --dataset-with-size-type "Medium Cohere (768dim, 1M)" \
  --db-label ydb-3n --drop-old --load --search-serial --search-concurrent
```

> См. §6, недоработка C: для int-фильтра префикс индекса — это уникальный `id`,
> и диапазонный предикат `id >= …` плохо ложится на «фильтрованный» kmeans-tree.
> Этот блок плана нужно валидировать на recall и на отсутствие full-scan
> (по статистике запроса), прежде чем публиковать цифры.

### Фаза 5 — Кривая recall ↔ QPS (тюнинг клиента)

Для каждой СУБД лидерборд строит кривую «recall vs QPS». Для YDB её даёт свип
`--kmeans-tree-search-top-size` (и при необходимости `--levels`/`--clusters`):

```shell
for TOP in 1 4 10 16 32; do
  vectordbbench ydb --case-type Performance768D1M --db-label "ydb-3n-top$TOP" \
    --skip-drop-old --skip-load --search-serial --search-concurrent \
    --num-concurrency 1,10,50,80 --concurrency-duration 30 \
    --kmeans-tree-search-top-size $TOP
done
```

Так как данные и индекс не меняются, грузим один раз (`--drop-old --load` на top=10),
дальше только варьируем поиск с `--skip-drop-old --skip-load`.

### Фаза 6 — Конкуренты на тех же кейсах

Повторяем фазы 1/3/4 для каждого конкурента с теми же `--case-type`, `--k`,
`--num-concurrency`, `--concurrency-duration`. Пример для pgvector:

```shell
vectordbbench pgvectorhnsw --case-type Performance768D1M --db-label pg-hnsw \
  --user-name ... --password ... --host ... --db-name ... \
  --m 16 --ef-construction 128 --ef-search 128 \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30
```

Удобно собрать все системы в **batch-конфиг** и запустить одной командой
`vectordbbench batchcli --batch-config-file plan.yaml` (формат см. в README).

### Сводка порядка

1. Фаза 0 (smoke) → 2. Фаза 1 (baseline 1 узел) → 3. Фаза 2 (1/3/9 узлов на 10M)
→ 4. Фаза 3 (объём) → 5. Фаза 4 (фильтры) → 6. Фаза 5 (кривая recall/QPS)
→ 7. Фаза 6 (конкуренты) → 8. Сборка отчёта в UI (`init_bench`) и сравнение.

## 5. Достаточно ли кода для этого плана?

Что **уже работает** и закрывает план:

- **CLI и регистрация клиента.** Команда `vectordbbench ydb` зарегистрирована
  (`cli/vectordbbench.py`), `DB.YDB` добавлен в enum, `init_cls`/`config_cls`/
  `case_config_cls` (`backend/clients/__init__.py`). YDB также попадает в список
  СУБД веб-UI (`DB_LIST = [d for d in DB if d != DB.Test]`) и имеет блок
  параметров в `frontend/config/dbCaseConfigs.py`.
- **Полный жизненный цикл кейса.** Реализованы `init`, `insert_embeddings`,
  `optimize` (построение `vector_kmeans_tree` **после** загрузки), `search_embedding`,
  `prepare_filter`. Это покрывает Load + Search(serial/concurrent).
- **Метрика расстояния** подставляется автоматически: `assembler.py`
  присваивает `db_case_config.metric_type = dataset.metric_type`, а
  `YDBIndexConfig` корректно маппит это в `cosine/euclidean/inner_product` и в
  соответствующую `Knn::*`-функцию и порядок сортировки.
- **Конкурентный поиск масштабируется на стороне клиента.** Каждый процесс
  `MultiProcessingSearchRunner` открывает свой `init()` → свой `Driver` +
  `QuerySessionPool`, поэтому `--num-concurrency` отрабатывает корректно
  (это и есть нагрузочная ось для фаз 2/5). `thread_safe = True`.
- **Изоляция кейсов по таблицам.** `task_runner._case_unique_collection_name`
  даёт YDB (как и Doris) уникальное имя таблицы на кейс; есть и явный
  `--table-name`. Это позволяет держать несколько кейсов/прогонов рядом.
- **Параметры индекса/поиска вынесены в CLI и UI**: `--levels`, `--clusters`,
  `--kmeans-tree-search-top-size`, `--cover-embedding` — достаточно для фазы 5.
- **Авто-подбор формы дерева** (`compute_kmeans_tree_params`) масштабируется с
  размером датасета (1/2/3 уровня), что разумно для 50K…10M.
- **Ожидание готовности индекса** через пробный запрос к `VIEW index` —
  защищает от преждевременного старта поиска.
- **Тесты** (`tests/test_ydb.py`) покрывают конфиг, маппинг метрик, авторизацию,
  фильтры; есть интеграционный тест (скипается без живого YDB).

Вывод: **для базового сопоставимого сравнения (фазы 0, 1, 3, 6 без фильтров,
частично 5) кода достаточно.** Однако для центральной части задачи —
демонстрации **горизонтального масштабирования** (фаза 2) и корректной
фильтрации (фаза 4) — есть существенные пробелы (см. §6).

## 6. Недоработки (приоритезированный список)

### G (блокер для фазы масштабирования). Нет управления партиционированием/шардированием таблицы

`YDB._create_table` создаёт таблицу так:

```sql
CREATE TABLE ... ( id Uint64 NOT NULL, embedding String NOT NULL, PRIMARY KEY (id) );
```

— **без** настроек `AUTO_PARTITIONING_*` / `UNIFORM_PARTITIONS` /
`PARTITION_AT_KEYS`. По умолчанию строковая таблица YDB стартует как **одна
таблетка** (split по размеру включён, но по нагрузке `AUTO_PARTITIONING_BY_LOAD`
выключен). На небольших/средних датасетах таблица может остаться в 1–2 таблетках
и физически жить на одном-двух узлах, из-за чего **прирост от добавления узлов в
поиске не проявится** — а это и есть суть фазы 2.

Рекомендация: добавить настраиваемые опции и проставлять их при создании таблицы,
например:

```sql
WITH (
  AUTO_PARTITIONING_BY_LOAD = ENABLED,
  AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = <N>,
  AUTO_PARTITIONING_PARTITION_SIZE_MB = <...>,
  UNIFORM_PARTITIONS = <N>      -- предсоздание N шардов
)
```

и вынести их в CLI (`--min-partitions`, `--auto-partition-by-load`,
`--uniform-partitions`). Без этого фазу 2 придётся выполнять, вручную
донастраивая таблицу вне бенчмарка (что ломает воспроизводимость).

### H (важно для масштабирования записи). Последовательный `Uint64` первичный ключ → горячий шард

`id` грузится монотонно возрастающим (`metadata = range(...)`). При
`UNIFORM_PARTITIONS` диапазон ключей делится равномерно, но монотонные ключи
заполняют партиции **по очереди** → во время загрузки активна одна «хвостовая»
таблетка (классический горячий шард). Это занижает throughput вставки и мешает
показать масштабирование записи (фаза 2/CloudInsert).

Рекомендация: рассмотреть хэширование/перестановку ключа (например, PK по
бит-реверсу id или `(Digest::...(id), id)`), либо явно документировать
ограничение. Минимум — сделать оговорку в плане при интерпретации
`insert_duration`.

### C (важно для фазы 4). Int-фильтр строит индекс с префиксом по уникальному `id`

Для `FilterOp.NumGE` `index_on_columns` возвращает `("id", "embedding")`, а
предикат — диапазон `WHERE id >= value`. По документации YDB «фильтрованный»
`vector_kmeans_tree` рассчитан на **равенство** по префиксным колонкам
(`WHERE user = 'john'`); префикс по **уникальному** `id` вырождает структуру, а
диапазонный предикат плохо отображается на префиксно-фильтрованный индекс.
Следствие — риск низкого recall или фактического full-scan на кейсах
`*1P/*99P`.

Рекомендация: проверить план/статистику запроса на этих кейсах; при
подтверждении проблемы — либо хранить отдельную фильтр-колонку с низкой
кардинальностью, либо честно пометить int-фильтр-кейсы как неподдерживаемые/
экспериментальные для YDB. Label-фильтр (равенство по `labels`) к индексу
ложится корректно.

### D (важно для больших датасетов). Таймаут ожидания индекса захардкожен и оторван от `optimize_timeout`

`YDB_INDEX_WAIT_TIMEOUT_SECONDS = 7200` (2 часа) фиксирован в `ydb_client.py`,
тогда как кейс задаёт свой `optimize_timeout` (для 10M — 10 суток). Построение
индекса на 5M/10M может занять **дольше 2 часов**, и `_wait_for_index` упадёт по
`TimeoutError` раньше, чем сработает легитимный таймаут кейса. Это прямо мешает
фазам 2 и 3 на 10M.

Рекомендация: пробрасывать `optimize_timeout` (доступен как
`data_size`/контекст кейса) в ожидание индекса либо сделать таймаут
конфигурируемым (`--index-build-timeout`), по умолчанию синхронизировать с
`optimize_timeout`.

### E (средне). Размер батча вставки захардкожен (1000), `NUM_PER_BATCH` игнорируется

В `insert_embeddings` `batch_size = 1000` зашит в код; глобальный
`config.NUM_PER_BATCH` и переменная окружения не учитываются. Для тюнинга
throughput вставки (важно при сравнении масштабирования записи) это ограничивает
гибкость.

Рекомендация: брать размер батча из `NUM_PER_BATCH`/CLI-опции.

### F (средне). Размер `QuerySessionPool` не настраивается

`ydb.QuerySessionPool(driver)` создаётся с дефолтным размером. При высоких
конкуррентностях в рамках одного процесса (или при будущем переходе на
тред-модель) это может стать узким местом. Сейчас за счёт процессной модели не
критично, но стоит вынести в опцию (`--session-pool-size`).

### B (мелко, UX). README обещает кейсы, чувствительные к §C/§D

В README раздел «Run YDB» перечисляет int-фильтр-кейсы (`*1P/*99P`) как
поддерживаемые. До устранения недоработки C их стоит пометить как требующие
валидации, чтобы не публиковать некорректные цифры.

### A (мелко). Нет явного контроля метрики на стороне YDB-CLI

Метрика берётся из датасета через `assembler` — это правильно для штатных
кейсов. Но для кастомных датасетов/ручных прогонов отсутствует CLI-override
метрики у YDB (у части других клиентов он есть). Не блокер для плана.

## 7. Итог

- **Сопоставимое сравнение YDB ↔ конкуренты** на безфильтровых кейсах
  (фазы 0/1/3/5/6) текущая реализация поддерживает.
- Для **главной темы задачи — горизонтального масштабирования (фаза 2)** —
  нужен минимум доработок **G** (партиционирование таблицы) и **D** (таймаут
  построения индекса на больших датасетах); желательно **H** (горячий шард при
  вставке).
- Для **фильтрации (фаза 4)** обязательна валидация **C** (int-фильтр на
  префиксе по уникальному id), label-фильтр работоспособен.

Рекомендуемый порядок устранения: **G → D → C → H → E → F → B → A**.
