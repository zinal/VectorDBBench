# План тестирования YDB в VectorDBBench и анализ достаточности кода

Документ описывает конкретный план запуска бенчмарков векторного поиска для
сравнения **YDB** с популярными векторными СУБД, с акцентом на **горизонтальное
масштабирование YDB**. Вторая часть документа — проверка того, что текущая
реализация поддержки YDB (`vectordb_bench/backend/clients/ydb/`) достаточна для
выполнения этого плана, и перечень обнаруженных недоработок.

> Актуально для ветки `ydb-support-initial` по состоянию на ревизию `5e5112d`
> (коммиты: авто-партиционирование таблицы и индекса, опция `--overlap-clusters`,
> регистрация YDB в UI, корректная фильтрация по меткам).

---

## 1. Цели и методология

YDB — распределённая СУБД, и её главное отличие от большинства движков из
лидерборда (pgvector, Qdrant, Milvus standalone, Elasticsearch single-node и
т. п.) — это **горизонтальное масштабирование**: данные разбиваются на таблетки
(DataShard), которые распределяются по узлам кластера, а SDK-драйвер сам
балансирует запросы. Поэтому план строится вокруг двух осей:

1. **Сопоставимость** — YDB и конкуренты прогоняются на одинаковых датасетах,
   кейсах и при одинаковой нагрузке клиента (одни и те же `--case-type`, `--k`,
   `--num-concurrency`, `--concurrency-duration`), чтобы результаты можно было
   класть рядом на лидерборде и сравнивать по QPS, p99-latency и recall.
2. **Масштабируемость** — YDB прогоняется на кластерах **разного размера**
   (например 1 → 3 → 9 узлов) на одном и том же кейсе, чтобы измерить, как
   растут QPS/throughput и как ведёт себя latency и recall при добавлении узлов.

Ключевой принцип VectorDBBench: «система под тестом разворачивается как угодно,
клиентские параметры можно тюнить». Размер кластера YDB — это **операционный**
параметр развёртывания, его сам бенчмарк не задаёт; различия между прогонами
фиксируются через `--db-label` (рекомендуется кодировать туда размер/тип
инстанса, как это делается для эталонных результатов проекта).

### Метрики, которые собирает фреймворк (на кейс)

- `insert_duration`, `optimize_duration`, `load_duration` (загрузка + построение индекса);
- `recall`, `ndcg`, `serial_latency_p99/p95` (серийный поиск, 1 поток);
- `qps` (максимум по списку конкуррентностей), а также по каждой конкуррентности —
  `conc_qps_list`, `conc_latency_p99/p95/avg_list`.

### Базовые предпосылки

- Клиент бенчмарка во всех прогонах одинаковый (тот же хост/CPU/сеть), желательно
  в той же зоне, что и сервер — иначе сетевой RTT исказит latency.
- Метрика расстояния выбирается автоматически из датасета (см. §5, `assembler.py`),
  отдельно её задавать не нужно.
- Каждый кейс изолируется собственной таблицей (`--table-name` или
  авто-имя по case-type, см. `task_runner._case_unique_collection_name`).

## 2. Подготовка окружения

### 2.1. Конкуренты (минимальный сопоставимый набор)

Берём системы, которые уже есть в проекте и покрывают разные классы движков:

| Класс | СУБД | Команда CLI |
|------|------|-------------|
| PostgreSQL + HNSW | pgvector | `vectordbbench pgvectorhnsw` |
| PostgreSQL + IVF/RaBitQ | VectorChord | `vectordbbench vectorchordrq` |
| Специализированный ANN | Milvus | `vectordbbench milvushnsw` |
| Специализированный ANN | Qdrant | `vectordbbench qdrant` |
| Поисковый движок | Elasticsearch/OpenSearch | `vectordbbench elasticcloudhnsw` / `awsopensearch` |
| Распределённый OLTP/HTAP + вектор | **YDB** | `vectordbbench ydb` |

Для честного сравнения **«масштабируемость против масштабируемости»** в качестве
оппонентов берём системы, которые тоже умеют шардироваться: Milvus (distributed),
OpenSearch (shards/replicas), Qdrant (sharding). Single-node движки (pgvector)
оставляем как «однопроцессный baseline».

### 2.2. YDB кластеры

Готовим **минимум 3 конфигурации** одного и того же кластера, отличающиеся только
числом узлов хранения/вычисления, например:

- `ydb-1n` — 1 узел;
- `ydb-3n` — 3 узла;
- `ydb-9n` — 9 узлов.

Подключение во всех случаях — через endpoint discovery (драйвер сам балансирует):

```shell
export YDB_ENDPOINT=grpc://<lb-or-any-node>:2136
export YDB_DATABASE=/Root/db1
# при необходимости: YDB_USER / YDB_PASSWORD или SDK-креды
```

> Состояние на ревизии `5e5112d`: клиент **уже** создаёт таблицу с включённым
> авто-партиционированием (`AUTO_PARTITIONING_BY_SIZE = ENABLED`,
> `AUTO_PARTITIONING_BY_LOAD = ENABLED`, настраиваемые `MIN/MAX_PARTITIONS_COUNT`)
> и донастраивает внутренние таблицы индекса (`indexImpl*Table`) после
> построения. Это закрывает прежний блокер с одношардовой таблицей.
>
> Остаётся нюанс (см. §6, G′): `AUTO_PARTITIONING_MIN_PARTITIONS_COUNT` по
> документации YDB **не предсоздаёт** партиции, а лишь запрещает слияние ниже
> порога; эагерное предразбиение даёт только `UNIFORM_PARTITIONS`/
> `PARTITION_AT_KEYS`, которые в коде не используются. Поэтому таблица стартует с
> одного шарда и разрастается в фоне по размеру/нагрузке — отсюда требование к
> **прогреву** перед замером масштабирования (см. фазу 2).

Параметры партиционирования задаются на стороне подключения (одинаково для всех
прогонов одного кластера). Согласно рекомендации YDB, минимум партиций имеет
смысл держать **не ниже числа выделенных ядер CPU** базы; дефолт клиента
`--auto-partitioning-min-partitions-count 1000` рассчитан на крупные кластеры —
для маленьких (1–3 узла) его стоит уменьшить, например:

```shell
# пример для небольшого кластера
--auto-partitioning-min-partitions-count 50 \
--auto-partitioning-max-partitions-count 200 \
--auto-partitioning-partition-size-mb 1000
```

## 3. Матрица датасетов и кейсов

Используем штатные кейсы (все они уже заявлены как поддерживаемые для YDB):

| Ось | Кейсы | Назначение |
|-----|-------|-----------|
| Размер (768d, Cohere) | `Performance768D1M` → `Performance768D10M` | масштаб по объёму |
| Размер (1536d, OpenAI) | `Performance1536D500K` → `Performance1536D5M` | большая размерность |
| Малый (для smoke) | `Performance1536D50K` | быстрый прогон/отладка |
| Int-фильтр | `Performance768D1M1P` / `...99P`, `Performance1536D500K1P` / `...99P` | фильтрация по диапазону id |
| Label-фильтр | `LabelFilterPerformanceCase` (+ `--dataset-with-size-type`) | фильтрация по равенству метки |

Свип параметров YDB-индекса для кривой recall/QPS:
`--kmeans-tree-search-top-size` ∈ {1, 4, 10, 16, 32}, `--overlap-clusters` ∈ {1, 3, 5},
опционально `--levels`/`--clusters`.

## 4. Конкретный порядок запуска

Везде ниже подразумевается активированное окружение
(`pip install -e '.[ydb]'`) и выставленные `YDB_ENDPOINT`/`YDB_DATABASE`.
Конкуррентность и длительность держим **одинаковыми** для YDB и конкурентов.

### Фаза 0 — Smoke / валидация конвейера (минуты)

Цель: убедиться, что загрузка → построение индекса → поиск работают end-to-end,
прежде чем тратить часы на большие датасеты.

```shell
vectordbbench ydb \
  --case-type Performance1536D50K \
  --db-label ydb-1n-smoke \
  --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,10 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10 \
  --auto-partitioning-min-partitions-count 8 --auto-partitioning-max-partitions-count 64
```

Проверяем в результатах: `recall` разумный (> 0.9 при top-size 10 на 50K),
`load_duration` конечный, `qps` ненулевой. Также прогнать unit/инт-тесты:
`pytest tests/test_ydb.py` (интеграционные требуют живого YDB).

### Фаза 1 — Однокластерный baseline (medium датасеты)

Цель: получить опорную точку YDB на 1 узле для последующего сравнения с
конкурентами и с многоузловыми прогонами.

```shell
# 768d / 1M (Cohere)
vectordbbench ydb --case-type Performance768D1M \
  --db-label ydb-1n --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10

# 1536d / 500K (OpenAI)
vectordbbench ydb --case-type Performance1536D500K \
  --db-label ydb-1n --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30 \
  --kmeans-tree-search-top-size 10
```

### Фаза 2 — Горизонтальное масштабирование (главная фаза для YDB)

Цель: на **одном и том же кейсе** замерить прирост QPS/latency при росте числа
узлов. Меняется только кластер и `--db-label`; всё остальное идентично.

```shell
# Кластер 1 узел
vectordbbench ydb --case-type Performance768D10M --db-label ydb-1n \
  --drop-old --load --search-serial --search-concurrent \
  --num-concurrency 1,10,20,40,60,80,100,120 --concurrency-duration 60 \
  --kmeans-tree-search-top-size 10

# Кластер 3 узла (переключить YDB_ENDPOINT/кластер) — те же параметры
vectordbbench ydb --case-type Performance768D10M --db-label ydb-3n \
  --skip-drop-old --skip-load --search-serial --search-concurrent \
  --num-concurrency 1,10,20,40,60,80,100,120 --concurrency-duration 60 \
  --kmeans-tree-search-top-size 10

# Кластер 9 узлов — те же параметры, --db-label ydb-9n
```

Замечания (важно для корректности именно фазы масштабирования):
- **Прогрев обязателен.** Поскольку партиции **не предсоздаются**
  (`UNIFORM_PARTITIONS` не используется, см. §6 G′), и таблица данных, и
  внутренние таблицы индекса стартуют с одного шарда и разъезжаются по узлам
  только по мере роста размера/нагрузки. «Холодный» замер сразу после загрузки
  занизит масштабирование. Поэтому перед зачётным прогоном дайте поисковую
  нагрузку «на прогрев» (например, отдельный предварительный
  `--search-concurrent` с высокой конкуррентностью), дождитесь стабилизации
  числа партиций/распределения по узлам (видно в мониторинге YDB), и только
  потом снимайте зачётные числа.
- На фазе масштабирования данные грузим **один раз на кластер** (`--drop-old --load`
  на первом запуске для данного кластера), а повторные конфигурации
  поиска запускаем с `--skip-drop-old --skip-load`.
- Доводим список конкуррентностей до значений, где QPS выходит на плато — именно
  «потолок» при разном числе узлов показывает масштабируемость.
- Для теста масштабирования **записи/throughput** отдельно снимаем
  `insert_duration` на 1/3/9 узлах. Учитывайте недоработку H (§6): монотонный
  `Uint64`-ключ создаёт «горячий» хвостовой шард на старте загрузки, поэтому
  прирост throughput записи от добавления узлов может быть занижен.

### Фаза 3 — Масштабирование по объёму данных

Цель: показать, как YDB держит рост датасета (фиксируем кластер, например 3 узла).

```shell
for CASE in Performance768D1M Performance768D10M; do
  vectordbbench ydb --case-type $CASE --db-label ydb-3n \
    --drop-old --load --search-serial --search-concurrent \
    --num-concurrency 1,10,20,40,60,80 --concurrency-duration 60 \
    --kmeans-tree-search-top-size 10
done
```

> Для `Performance768D10M`/`Performance1536D5M` построение индекса может быть
> долгим. Сейчас ожидание готовности индекса захардкожено на 2 часа
> (недоработка D, §6) — на больших датасетах прогон может упасть по таймауту
> раньше легитимного `optimize_timeout`. До исправления D держите эти кейсы под
> контролем и при необходимости патчите `YDB_INDEX_WAIT_TIMEOUT_SECONDS`.

### Фаза 4 — Фильтрация

```shell
# Int-фильтр (диапазон id): 1% и 99%
vectordbbench ydb --case-type Performance768D1M1P  --db-label ydb-3n --drop-old --load --search-serial --search-concurrent
vectordbbench ydb --case-type Performance768D1M99P --db-label ydb-3n --drop-old --load --search-serial --search-concurrent

# Label-фильтр (равенство метки) — нужен --dataset-with-size-type
vectordbbench ydb --case-type LabelFilterPerformanceCase \
  --dataset-with-size-type "Medium Cohere (768dim, 1M)" \
  --db-label ydb-3n --drop-old --load --search-serial --search-concurrent
```

> Label-фильтр теперь реализован корректно: при наличии скалярных меток индекс
> строится `ON (labels, embedding)` с `COVER (embedding)`, а поиск использует
> `WHERE labels = $label` (равенство по префиксу — штатный режим фильтрованного
> vector_kmeans_tree).
>
> Int-фильтр остаётся под вопросом (недоработка C, §6): префикс индекса — это
> **уникальный** `id`, а предикат — диапазон `id >= …`, что плохо ложится на
> префиксно-фильтрованный kmeans-tree. Перед публикацией цифр обязательно
> проверьте recall и статистику запроса (отсутствие full-scan главной таблицы).

### Фаза 5 — Кривая recall ↔ QPS (тюнинг клиента)

Для каждой СУБД лидерборд строит кривую «recall vs QPS». Для YDB её даёт свип
`--kmeans-tree-search-top-size` (и при необходимости `--overlap-clusters`,
`--levels`/`--clusters`):

```shell
for TOP in 1 4 10 16 32; do
  vectordbbench ydb --case-type Performance768D1M --db-label "ydb-3n-top$TOP" \
    --skip-drop-old --skip-load --search-serial --search-concurrent \
    --num-concurrency 1,10,50,80 --concurrency-duration 30 \
    --kmeans-tree-search-top-size $TOP
done
```

Так как данные и индекс не меняются, грузим один раз (`--drop-old --load` на top=10),
дальше только варьируем поиск с `--skip-drop-old --skip-load`. Замечание:
`--overlap-clusters` влияет на **построение** индекса, поэтому его свип требует
перестроения (отдельные прогоны с `--drop-old --load`), в отличие от
`--kmeans-tree-search-top-size`, который меняется только на поиске.

### Фаза 6 — Конкуренты на тех же кейсах

Повторяем фазы 1/3/4 для каждого конкурента с теми же `--case-type`, `--k`,
`--num-concurrency`, `--concurrency-duration`. Пример для pgvector:

```shell
vectordbbench pgvectorhnsw --case-type Performance768D1M --db-label pg-hnsw \
  --user-name ... --password ... --host ... --db-name ... \
  --m 16 --ef-construction 128 --ef-search 128 \
  --num-concurrency 1,5,10,20,30,40,60,80 --concurrency-duration 30
```

Удобно собрать все системы в **batch-конфиг** и запустить одной командой
`vectordbbench batchcli --batch-config-file plan.yaml` (формат см. в README).

### Сводка порядка

1. Фаза 0 (smoke) → 2. Фаза 1 (baseline 1 узел) → 3. Фаза 2 (1/3/9 узлов на 10M,
с прогревом) → 4. Фаза 3 (объём) → 5. Фаза 4 (фильтры) → 6. Фаза 5 (кривая
recall/QPS) → 7. Фаза 6 (конкуренты) → 8. Сборка отчёта в UI (`init_bench`) и
сравнение.

## 5. Достаточно ли кода для этого плана?

Что **уже работает** и закрывает план (ревизия `5e5112d`):

- **CLI и регистрация клиента.** Команда `vectordbbench ydb` зарегистрирована
  (`cli/vectordbbench.py`), `DB.YDB` добавлен в enum, `init_cls`/`config_cls`/
  `case_config_cls` (`backend/clients/__init__.py`). YDB присутствует в списке
  СУБД веб-UI, имеет иконку/цвет (`frontend/config/styles.py`) и блок параметров
  кейса в `frontend/config/dbCaseConfigs.py` (levels, clusters, top-size,
  overlap-clusters, cover-embedding).
- **Полный жизненный цикл кейса.** Реализованы `init`, `insert_embeddings`,
  `optimize` (построение `vector_kmeans_tree` **после** загрузки), `search_embedding`,
  `prepare_filter`. Это покрывает Load + Search(serial/concurrent).
- **Партиционирование/шардирование (НОВОЕ).** Таблица данных создаётся
  `WITH (AUTO_PARTITIONING_BY_SIZE = ENABLED, AUTO_PARTITIONING_BY_LOAD = ENABLED,
  AUTO_PARTITIONING_MIN/MAX_PARTITIONS_COUNT = …)`; после построения индекса его
  внутренние таблицы (`indexImplLevelTable`/`PostingTable`/`PrefixTable`)
  донастраиваются тем же ALTER с `AUTO_PARTITIONING_PARTITION_SIZE_MB`. Это
  основной механизм, позволяющий распределить нагрузку по узлам кластера —
  ключевое условие фазы 2.
- **Метрика расстояния** подставляется автоматически: `assembler.py`
  присваивает `db_case_config.metric_type = dataset.metric_type`, а
  `YDBIndexConfig` корректно маппит это в `cosine/euclidean/inner_product`.
- **Конкурентный поиск масштабируется на стороне клиента.** Каждый процесс
  `MultiProcessingSearchRunner` открывает свой `init()` → свой `Driver` +
  `QuerySessionPool`, поэтому `--num-concurrency` отрабатывает корректно.
- **Изоляция кейсов по таблицам** (`task_runner._case_unique_collection_name`) и
  явный `--table-name`.
- **Параметры индекса/поиска вынесены в CLI и UI**: `--levels`, `--clusters`,
  `--kmeans-tree-search-top-size`, `--overlap-clusters`, `--cover-embedding`,
  `--auto-partitioning-*` — достаточно для фаз 2 и 5.
- **Корректный label-фильтр**: при `with_scalar_labels` индекс `ON (labels,
  embedding)` + `COVER (embedding)`, поиск по равенству метки.
- **Авто-подбор формы дерева** (`compute_kmeans_tree_params`) масштабируется с
  размером датасета; пустые/нулевые `levels`/`clusters` трактуются как «авто».
- **Ожидание готовности индекса** через пробный запрос к `VIEW index`.
- **Тесты** (`tests/test_ydb.py`) покрывают конфиг, маппинг метрик, авторизацию,
  фильтры, границы партиционирования; есть интеграционный тест.

Вывод: по сравнению с предыдущей ревизией **главный блокер масштабирования
(отсутствие партиционирования) устранён**. Для базового сопоставимого сравнения и
для фазы 2 кода достаточно при соблюдении процедуры прогрева. Открытыми остаются
несколько недоработок (см. §6), наиболее значимые из которых — таймаут построения
индекса на больших датасетах (D) и корректность int-фильтра (C).

Прежний блокер закрыт коммитами «YDB table auto-partitioning setup» и «YDB index
auto-partitioning configured»: таблица и внутренние таблицы индекса создаются/
донастраиваются с `AUTO_PARTITIONING_BY_SIZE/BY_LOAD = ENABLED` и
настраиваемыми `MIN/MAX_PARTITIONS_COUNT`, `PARTITION_SIZE_MB`. Параметры
выведены в CLI (`--auto-partitioning-*`).

### G′ (важно для фазы 2). Нет начального предразбиения (`UNIFORM_PARTITIONS`)

По документации YDB `AUTO_PARTITIONING_MIN_PARTITIONS_COUNT` **не создаёт**
партиции заранее — он лишь запрещает слияние ниже порога. Эагерное предразбиение
даёт только `UNIFORM_PARTITIONS`/`PARTITION_AT_KEYS`, которые в коде не
используются. Следствия:
- таблица данных и индекс стартуют с одного шарда → «холодный» прогон занижает
  масштабирование, пока фоновое разбиение по размеру/нагрузке не разнесёт шарды
  по узлам (отсюда требование прогрева в фазе 2);
- к таблице **данных** не применяется `AUTO_PARTITIONING_PARTITION_SIZE_MB`
  (он задаётся только для внутренних таблиц индекса) → таблица данных делится по
  дефолтному порогу 2 ГБ, то есть крупно.

Рекомендация: добавить опцию `--uniform-partitions N` (и/или применять
`AUTO_PARTITIONING_PARTITION_SIZE_MB` к таблице данных), чтобы при создании сразу
разбивать таблицу на N шардов по диапазону `Uint64`-ключа. Это убирает
необходимость долгого прогрева и делает фазу 2 воспроизводимой. Минимум — описать
процедуру прогрева в инструкции (сделано в §4, фаза 2).

### D (важно для больших датасетов). Таймаут ожидания индекса захардкожен и оторван от `optimize_timeout`

`YDB_INDEX_WAIT_TIMEOUT_SECONDS = 7200` (2 часа) по-прежнему фиксирован в
`ydb_client.py`, тогда как кейс задаёт свой `optimize_timeout` (для 10M — 10
суток). Построение индекса на 5M/10M может занять дольше 2 часов, и
`_wait_for_index` упадёт по `TimeoutError` раньше легитимного таймаута кейса.
Прямо мешает фазам 2 и 3 на больших датасетах.

Рекомендация: пробрасывать `optimize_timeout` в ожидание индекса либо сделать
таймаут конфигурируемым (`--index-build-timeout`), по умолчанию синхронизировать с
`optimize_timeout`.

### C (важно для фазы 4). Int-фильтр строит индекс с префиксом по уникальному `id`

Для `FilterOp.NumGE` `index_on_columns` возвращает `("id", "embedding")`, а
предикат — диапазон `WHERE id >= value`. По документации YDB фильтрованный
`vector_kmeans_tree` рассчитан на **равенство** по префиксным колонкам;
префикс по **уникальному** `id` вырождает структуру, а диапазонный предикат плохо
отображается на префиксно-фильтрованный индекс. Риск низкого recall или
фактического full-scan на кейсах `*1P/*99P`. (Label-фильтр этой проблемы лишён —
там префикс `labels` низкой кардинальности и равенство.)

Рекомендация: проверить план/статистику запроса на int-фильтр-кейсах; при
подтверждении проблемы — либо хранить отдельную фильтр-колонку низкой
кардинальности, либо честно пометить int-фильтр-кейсы как
экспериментальные для YDB.

### H (средне, влияет на масштабирование записи). Последовательный `Uint64` PK → горячий шард на загрузке

`id` грузится монотонно (`metadata = range(...)`). Без `UNIFORM_PARTITIONS`
таблица стартует с одного шарда, и монотонный ключ держит активной одну
«хвостовую» таблетку, пока сплит по размеру/нагрузке не сработает. Это занижает
throughput вставки и прирост от добавления узлов в тесте записи. С `BY_LOAD =
ENABLED` ситуация лучше, чем раньше, но проблема старта остаётся.

Рекомендация: рассмотреть предразбиение (`UNIFORM_PARTITIONS`, см. G′) и/или
хэширование ключа; минимум — оговорка при интерпретации `insert_duration`
(сделано в §4, фаза 2).

### E (средне). Размер батча вставки захардкожен (1000), `NUM_PER_BATCH` игнорируется

В `insert_embeddings` `batch_size = 1000` зашит в код; глобальный
`config.NUM_PER_BATCH` и переменная окружения не учитываются. Ограничивает
тюнинг throughput вставки при сравнении масштабирования записи.

Рекомендация: брать размер батча из `NUM_PER_BATCH`/CLI-опции.

### F (мелко). Размер `QuerySessionPool` не настраивается

`ydb.QuerySessionPool(driver)` создаётся с дефолтным размером. При процессной
модели поиска не критично, но стоит вынести в опцию (`--session-pool-size`) на
случай высокой конкуррентности внутри процесса.

### A (мелко). Нет CLI-override метрики у YDB

Метрика берётся из датасета через `assembler` — правильно для штатных кейсов; для
кастомных датасетов/ручных прогонов overrides метрики у YDB нет. Не блокер.

## 7. Итог (по состоянию на `5e5112d`)

- **Главный блокер масштабирования устранён**: партиционирование таблицы и
  индекса включено и настраивается из CLI/UI. Фаза 2 выполнима **при условии
  прогрева** (т. к. предразбиения `UNIFORM_PARTITIONS` нет — недоработка G′).
- **Сопоставимое сравнение YDB ↔ конкуренты** без фильтров и **label-фильтр**
  поддерживаются полностью.
- Перед большими прогонами (10M/5M) учитывать **D** (захардкоженный таймаут
  построения индекса) — это наиболее вероятная причина падения зачётного
  прогона.
- **Int-фильтр** требует валидации recall/плана (**C**) перед публикацией цифр.

Рекомендуемый порядок устранения оставшихся пунктов: **D → C → G′ → H → E → F → A**.
